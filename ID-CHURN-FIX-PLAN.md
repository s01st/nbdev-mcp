# Plan: Correct nbdev cell-ID churn

**Status:** proposal · **Date:** 2026-06-13 · **Repo:** `~/Projects/NBDevMCP` (the `nbdev_mcp` MCP server)

## 1. Problem & root cause

nbdev writes one comment marker per exported cell into every generated `.py`:

```
# %% ../../nbs/00_utils/12_nbformat.ipynb #<cell.id>
```

The marker id is the **raw `cell.id`** (nbdev `maker.py` → `write_cells`: `f" #{cell.id}" if cell.get('id') else ""`). nbformat v4.5 cells are *supposed* to carry a stable `id`, but **most notebooks on disk have no `id` field persisted**. When that happens:

1. `nbformat.read(...)` assigns a **random** id per cell, **in memory only** (the `MissingIDFieldWarning` — and nbformat warns it "will stop doing so in the future").
2. `nbdev_export` (CLI) re-reads the notebook → **fresh random ids every run** → every `# %%` marker changes → the generated `.py` diffs on every export even when nothing changed.
3. Worse: if a `.py` was hand-edited (rich content added directly), the next `nbdev_export` regenerates from the stale notebook and **reverts the hand edits** — the churn masks/destroys real content. This is exactly what happened to the sva MCP package (`~/Projects/sva/srcs/mcp/pkg`), recovered manually on 2026-06-13.

**Dogfood evidence — this very repo is a victim:** of `nbs/**/*.ipynb`, **550 of 790 cells (36 notebooks) have no persisted `id`**, including `nbs/00_utils/12_nbformat.ipynb` (34/34) and `nbs/11_tools/05_lint.ipynb` (26/26). So `nbdev_export` here is non-deterministic today.

**Root cause in one line:** notebook cells lack *persisted* `id` fields, so ids are random-per-read and nbdev's markers churn. The fix is to **persist stable cell ids** and make export **ensure** them first.

## 2. Why NBDevMCP is the right place to fix it

- NBDevMCP already **owns notebook I/O** (`nbdev_mcp/utils/nbformat.py` → `read_notebook`/`write_notebook`) and the **edit/add/read cell tools** (`nbdev_mcp/tools/editing.py`, which round-trip through those helpers).
- Its `nbdev_export` wrapper (`nbdev_mcp/tools/nbdev.py:95`) **shells out to the `nbdev_export`/`nbdev-export` CLI** — so we cannot (and should not) change nbdev's marker logic, but we **can persist stable ids in the source notebooks before the CLI reads them**.
- It has a **lint + autofix architecture** (`nbdev_mcp/tools/lint.py`: `lint_rules`, `lint_notebook_structure`, `auto_fix_lint:615`, `add_lint_tools`) — the idiomatic home for a detect-and-fix rule.

Today there is **zero** id handling anywhere in the package (confirmed by grep). This is greenfield.

> nbdev meta-note: NBDevMCP is itself an nbdev project. Every change below is made in the **source notebook under `nbs/`** (see each `.py`'s `AUTOGENERATED! File to edit:` header), then `nbdev_export`. Once the core helper lands, **dogfood it on this repo's own 36 notebooks first** (§7).

## 3. Design decision — id strategy

**Preserve-existing, fill-missing-deterministically, never-reassign.**

- **Preserve** any existing `cell.id` (never rewrite an id that's already there → stable identity across future edits, and never disturbs an already-clean notebook).
- **Fill** a missing id with a **deterministic** value so that (a) re-running the fix is idempotent and (b) two machines stamping the same un-stamped notebook produce the **same** ids (no cross-machine churn before commit).
  - id = `sha1(f"{nb_relpath}:{cell_index}:{cell_source}")[:8]`, sanitized to nbformat's allowed charset `[A-Za-z0-9_-]`, with a `-2`,`-3`… suffix on the rare intra-notebook collision (e.g. two identical empty `#| export` cells).
  - Rationale vs alternatives: pure nbformat `normalize()` would persist *random* ids (stops churn but not reproducible across machines); pure positional ids churn on insert. Content+index hash is deterministic for the one-time fill, and because we **preserve** thereafter, later inserts/edits keep existing ids and only brand-new cells get a fresh id.
- **New cells** created by NBDevMCP already get an id via `new_code_cell`/`new_markdown_cell`; keep that (they persist on write).

This mirrors exactly the manual sva recovery: ids stamped once, preserved forever, export becomes a deterministic no-op.

## 4. Implementation

### 4.1 Core helper — `nbs/00_utils/12_nbformat.ipynb` → `utils/nbformat.py`
- `ensure_cell_ids(nb, nb_relpath, *, overwrite=False) -> tuple[NotebookNode, int]` — fill missing (or all, if `overwrite`) ids deterministically per §3; return `(nb, n_changed)`.
- `stamp_notebook_ids(path, *, overwrite=False) -> int` — `read_notebook` → `ensure_cell_ids` → `write_notebook` only if changed; return count. (Use raw `json` read here, **not** `nbformat.read`, so we see truly-missing ids rather than nbformat's transient assignment.)
- Make **`write_notebook` id-safe**: call `ensure_cell_ids` before writing (so every notebook NBDevMCP writes is left clean). Keep `read_notebook` as-is but add the raw-aware path for detection.
- Export the two new symbols in `__all__`.

### 4.2 Prevent at export — `tools/nbdev.py` (`nbdev_export`, `nbdev_prepare`)
- Before the `run(cmd)` shell-out, run a **pre-export id pass**: stamp every `nbs/**/*.ipynb` in the project via `stamp_notebook_ids` (skip if clean). Add an opt-out `ensure_ids: bool = True` param.
- Include `{"ids_stamped": {nb: n, ...}}` in the returned dict so the caller sees what was persisted. This makes `nbdev_export` **idempotent** (markers stable across runs) for any project, fixing the churn at the point it bites.

### 4.3 Lint rule + autofix — `nbs/11_tools/05_lint.ipynb` → `tools/lint.py`
- New rule `cell_ids` in `lint_rules` / `lint_notebook_structure`: flag cells with **missing** ids and **duplicate** ids within a notebook; report `nb:cell_index`.
- Wire into `auto_fix_lint` (alongside the existing `no_relative_imports` fixer) to call `stamp_notebook_ids`.
- Surface counts in `check_notebook`.

### 4.4 Editing tools persist ids — `tools/editing.py`
- `add_notebook_cell`/`edit_notebook_cell` already `read_notebook → mutate → write_notebook` (`:513`/`:555`); once `write_notebook` is id-safe (§4.1) these persist ids automatically. Add an explicit `ensure_cell_ids` call after mutation for clarity, and ensure new cells keep their generated id.

### 4.5 Recovery tool — reconcile ids FROM an existing `.py` (the already-drifted case)
For projects whose **committed `.py` already has markers** that don't match the notebook (sva pkg case), naive stamping would change the `.py`. Add:
- `reconcile_ids_from_py(nb_path, py_path) -> int` — parse the committed `.py`'s ordered `# %% {path} #{id}` blocks, match to the notebook's export cells **by document order / defining symbol**, assign those exact ids to the cells, persist. Result: the notebook regenerates the **existing** `.py` byte-for-byte (zero diff) — recovery without churn.
- Expose as an MCP tool `reconcile_notebook_ids` (and/or a `--from-py` mode of the stamp tool). This is the generalized, reusable version of the manual sva fix.

## 5. Verification / tests — `nbs/99_tests/`
- **Determinism:** fixture notebook with no ids → `stamp` → `nbdev_export` twice → generated `.py` **byte-identical** across runs (the core regression).
- **Idempotency:** second `stamp` reports `0 changed`; ids unchanged.
- **Preserve:** a notebook with existing ids → `stamp` is a no-op (no reassignment).
- **Determinism cross-instance:** same un-stamped notebook stamped twice from scratch → identical ids.
- **Collision:** two identical-source cells → distinct ids (`-2` suffix), both valid per nbformat charset.
- **Recovery:** notebook + a hand-richer `.py` with mismatched markers → `reconcile_ids_from_py` → re-export → `.py` unchanged (`git diff` empty).
- **Validity:** stamped notebooks pass `nbformat.validate` with no `MissingIDFieldWarning`.

## 6. Rollout
1. Land §4.1 core helper (notebook + `nbdev_export`); add tests (§5).
2. **Dogfood:** `stamp` this repo's own 36 notebooks (550 missing ids) → commit the id-only notebook diff → confirm a subsequent `nbdev_export` is a no-op on `nbdev_mcp/**`.
3. Wire §4.2 prevention + §4.3 lint/autofix + §4.4 editing.
4. Ship §4.5 recovery tool; document the sva-pkg-style recovery in README.
5. Optional CI gate: `lint cell_ids` fails the build if any source notebook has missing/duplicate ids (prevents regression).

## 7. Risks & non-goals
- **Notebook diffs are expected** when first stamping (id-only churn, one time) — commit them deliberately, separate from content changes.
- **`__all__`/auto-block formatting** is a *separate* drift class (hand-formatted `__all__` won't byte-match nbdev's `TextWrapper@120` output — seen in sva `architecture.py`). Out of scope here; note it where relevant.
- **No change to nbdev itself** — we only persist ids and ensure them pre-export; nbdev's marker logic is untouched.
- **Don't reassign existing ids** — doing so would re-churn already-clean projects. Preserve is the invariant.
